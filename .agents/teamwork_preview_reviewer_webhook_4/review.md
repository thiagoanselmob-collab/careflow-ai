# Review Report: WhatsApp Webhook Implementation

## Review Summary

**Verdict**: REQUEST_CHANGES

The implementation of the WhatsApp Webhook is mostly functional and structured, but fails a key milestone (Milestone 1) and contains critical correctness, concurrency, and security issues. Specifically:
1. SQLite URI support is broken, which causes physical databases to be written to disk instead of being kept in-memory.
2. The concurrency test `test_concurrency_debounce_aggregation` fails because database state on disk is not isolated between test runs.
3. The Redis lock is vulnerable to race conditions due to non-unique lock values and lack of validation during release.
4. Raw SQL query string formatting is used for deleting message buffers, introducing a security/SQL injection smell.

---

## Findings

### [Critical] Finding 1: Broken SQLite URI Support (Milestone 1 Failure)

- **What**: The database manager does not pass `connect_args={"uri": True}` when calling `create_async_engine` for SQLite connection URIs.
- **Where**: `app/core/tenant_database.py`, line 155:
  ```python
  engine = create_async_engine(decrypted_conn_str, echo=False, future=True)
  ```
- **Why**: SQLite requires URI mode to be explicitly enabled in the driver to support connection strings like `sqlite+aiosqlite:///file:org_web_test?mode=memory&cache=shared`. Without `connect_args={"uri": True}`, SQLite treats `file:org_web_test` as a literal filename and creates a physical database file on disk. This directly violates **Milestone 1** of `PROJECT.md`.
- **Suggestion**: Update engine creation to pass `connect_args={"uri": True}` when the dialect is SQLite:
  ```python
  connect_args = {}
  if "sqlite" in decrypted_conn_str:
      connect_args["uri"] = True
  engine = create_async_engine(decrypted_conn_str, echo=False, future=True, connect_args=connect_args)
  ```

### [Critical] Finding 2: Concurrency Test Failure

- **What**: Running the test suite via `poetry run pytest` fails on the concurrency aggregation test `test_concurrency_debounce_aggregation`.
- **Where**: `tests/test_webhook_queue.py`, line 214:
  ```python
  assert called_state["messages"][-1].content == "Quero marcar consulta com o Dr. André Seabra"
  ```
- **Why**: Because of Finding 1, physical database files persist on disk between test runs. Leftover database entries in `message_buffer` and `dados_cliente` contaminate subsequent test executions, causing duplicate message aggregation and causing the test assertions to fail.
- **Suggestion**: Resolving Finding 1 will ensure that in-memory databases are isolated and destroyed on connection close, fixing this test failure.

### [Major] Finding 3: Redis Lock Concurrency Race Condition

- **What**: The Redis mutex lock uses a static value `"locked"` and is deleted blindly in the `finally` block.
- **Where**: `app/api/webhook.py`, lines 107 and 218:
  ```python
  lock_acquired = await redis_client.set(lock_key, "locked", nx=True, ex=10)
  ...
  finally:
      await redis_client.delete(lock_key)
  ```
- **Why**: If a task execution (e.g. executing LangGraph via `graph.invoke`) takes longer than the 10-second lock TTL, the lock will expire. A new concurrent task can then acquire the lock. When the first task finishes, it will call `redis_client.delete(lock_key)` in its `finally` block, releasing the lock currently held by the second task. This exposes the system to concurrent execution without lock protection.
- **Suggestion**: Use a unique identifier (like a UUID) as the lock value, and release the lock using a Lua script that verifies the value matches before deletion:
  ```python
  # Acquire
  lock_value = str(uuid.uuid4())
  lock_acquired = await redis_client.set(lock_key, lock_value, nx=True, ex=30)
  ...
  # Release Lua Script
  lua_release = """
  if redis.call("get", KEYS[1]) == ARGV[1] then
      return redis.call("del", KEYS[1])
  else
      return 0
  end
  """
  await redis_client.eval(lua_release, 1, lock_key, lock_value)
  ```

### [Major] Finding 4: Raw SQL String Formatting Smell

- **What**: The message deletion query uses python string formatting `','.join(map(str, message_ids))` to inject raw ID values directly into the SQL query.
- **Where**: `app/api/webhook.py`, line 133:
  ```python
  delete_query = text(f"DELETE FROM message_buffer WHERE id IN ({','.join(map(str, message_ids))})")
  ```
- **Why**: Injecting variables via f-strings into SQL statements is a security risk (potential SQL injection) and violates clean code standards. Even though `message_ids` are generated by the database, this pattern should be avoided in favor of safe parameter binding or SQLAlchemy ORM/expression constructs.
- **Suggestion**: Rewrite using SQLAlchemy ORM or binding:
  ```python
  # Using SQLAlchemy parameter binding:
  await session.execute(
      text("DELETE FROM message_buffer WHERE id = ANY(:ids)"), # PostgreSQL
      # Or standard expand parameters:
      text("DELETE FROM message_buffer WHERE id IN :ids"), 
      {"ids": tuple(message_ids)}
  )
  ```

### [Minor] Finding 5: Inefficient Sequential Session Management

- **What**: The background task redundantly opens and closes database sessions multiple times sequentially.
- **Where**: `app/api/webhook.py`, lines 114 and 138:
  ```python
  async with await tenant_db_manager.get_tenant_session(organization_id) as session:
      # read and consolidate
  ...
  async with await tenant_db_manager.get_tenant_session(organization_id) as session:
      # fetch or create ClientData
  ```
- **Why**: Re-acquiring a new database session increases transaction overhead and connection pooling utilization.
- **Suggestion**: Use a single async session block for all database actions in the task.

### [Minor] Finding 6: Logging Noise on WhatsApp Status Webhooks

- **What**: Standard WhatsApp API status updates (sent, delivered, read) trigger warnings in the log.
- **Where**: `app/api/webhook.py`, line 66:
  ```python
  logger.warning(f"Unprocessable webhook payload received: {payload}")
  ```
- **Why**: Status updates do not contain the `messages` array in the standard structure, returning `None` for `phone_number`. This outputs a warning and returns "unsupported payload format", which spams the logs with false alarms for normal traffic.
- **Suggestion**: Identify status payloads explicitly (e.g. checking if `"statuses"` is present in the payload) and ignore/return them gracefully without logging a warning.
